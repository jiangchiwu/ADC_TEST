# Debug: channel time delta

Status: [OPEN]

## Symptom

不同通道触发事件帧上送的时间差，与示波器观察到的 ADC 引脚实际信号时间差不一致。

## Evidence plan

本工程是 MCU 固件，无法直接通过 HTTP 日志上报运行时数据；本次使用已有 J-Link + SRAM4 诊断快照方式采集运行证据，并在必要时只增加 SRAM4 诊断字段作为 instrumentation。

## Hypotheses

1. H1: 当前事件时间用统一 `fr->start_time_us`，但 ADC1/ADC2/ADC3 DMA 半帧完成时间并不完全一致，导致不同 ADC 实例的通道使用了错误时间原点。
2. H2: DMA 半帧起始时间计算使用了错误的 half/full ISR 时间戳，导致半区边界附近事件的绝对时间偏移。
3. H3: 波峰/波谷定位只返回峰谷样本序号，但没有补偿 ADC rank 交织采样的通道内采样偏移，导致同 ADC 内两个 rank 存在固定采样相位误差。
4. H4: “第一个波峰/波谷”搜索从阈值点后开始，但阈值点可能受噪声/窗口判定影响提前或滞后，导致选到的不是示波器关注的那个首个有效峰谷。
5. H5: PC 端显示的 GAP 计算方式与固件事件时间定义不一致，导致视觉上看起来时间差不一致。

## Current step

Collected source evidence and runtime diagnostic evidence.

## Evidence

- Source evidence: `check_dma_and_push_frames()` used `adc1_dma_half_cyc` / `adc1_dma_full_cyc` to compute one shared `fr->start_time_us` for all 6 channels.
- Channel mapping evidence: CH1/CH2 come from ADC1, CH3/CH4 from ADC3, CH5/CH6 from ADC2.
- Runtime snapshot evidence from `diag_snapshot`:
  - half ISR cycles: ADC1=`0x1118883D`, ADC2=`0x11188B8A`, ADC3=`0x11188554`
  - full ISR cycles: ADC1=`0x1116669F`, ADC2=`0x111669A2`, ADC3=`0x1116637F`
  - half delta vs ADC1: ADC2=`+0x34D` cycles ≈ `+1.76us`, ADC3=`-0x2E9` cycles ≈ `-1.55us`
  - full delta vs ADC1: ADC2=`+0x303` cycles ≈ `+1.61us`, ADC3=`-0x320` cycles ≈ `-1.67us`
- Conclusion: H1 confirmed. Different ADC instances cannot share ADC1 frame time origin.
- H3 partially confirmed by design: rank2 within each 2-rank ADC scan is sampled after rank1, so rank2 needs half-sample-period compensation for sub-us precision.

## Fix plan

- Store per-channel DMA frame start time in `adc_frame_t`.
- Use ADC1 start time for CH1/CH2, ADC3 start time for CH3/CH4, ADC2 start time for CH5/CH6.
- Add `+0.5 sample` time compensation for rank2 channels CH2/CH4/CH6.

## Fix implemented

- `adc_frame_t` now stores `ch_start_time_us[ADC_NCH]`.
- `push_frame_from_dma()` receives per-channel start times instead of one shared start time.
- `Fill_ChannelStartTimes()` converts ADC1/ADC2/ADC3 DMA ISR timestamps to their own half-frame start times.
- Event timestamp now uses `fr->ch_start_time_us[ch] + event_us_in_frame`.

## Screenshot follow-up

User provided screenshot in `F:\work\ADC_TEST\test\测试波形.png`.

Additional finding:

- PC GUI originally calculated Event Stream `EVENT GAP` by serial receive order.
- Firmware may enqueue events in channel scan order CH1..CH6 for the same DMA frame, while true event order is determined by `TS_US`.
- Therefore PC `EVENT GAP` can differ from oscilloscope channel timing even when firmware timestamps are correct.

PC fix:

- Sort each batch of received events by `TS_US` before calculating global Event Stream gap.
- Keep channel card gap as same-channel adjacent-event interval.
- Updated `event_frame_protocol.md` to document this display semantics.
- Rebuilt `F:\ADC_FFT\test_scripts\dist\event_monitor_gui_frame12_115200.exe`.

## Screenshot 2 follow-up

User provided `F:\work\ADC_TEST\test\测试波形2.png`: oscilloscope CH1/CH2 interval is about `86us`, while PC showed about `705us`.

Additional evidence:

- CH1 and CH2 both belong to ADC1, so the previous cross-ADC time-origin issue cannot explain this case.
- SRAM4 snapshot before this fix showed firmware-side CH1/CH2 latest event timestamp low words differing by `0x2C1 = 705us`; therefore the 705us was generated by firmware timestamp selection, not by PC parsing.
- Code inspection found `Verify_Sustain_in_Frame()` rejected events near the DMA half-frame end because it required a full `SUSTAIN_SAMPLES=384` sample window after `trans_idx` inside the same half-frame.
- If CH2's real edge is close to the DMA half-frame boundary, it can fail sustain verification in that half-frame and be reported in a later half-frame, causing an apparent delay in the hundreds of microseconds.

Fix:

- `Verify_Sustain_in_Frame()` now uses an adaptive available window near the half-frame end.
- Hit-count and tail-hit requirements are scaled by the actually available sample count.
- It still requires `DEV_CONFIRM_CNT`, amplitude threshold, tail energy, and later FFT frequency validity, so it does not simply accept single-sample glitches.

## Screenshot 3 follow-up

User provided `F:\work\ADC_TEST\test\测试波形3.png`: oscilloscope CH1/CH2 envelope interval is about `253us`, while PC showed about `7us`. User also noted the signed order should be negative because CH2 occurs before CH1.

Additional evidence:

- CH1 and CH2 both belong to ADC1, so cross-ADC time-origin compensation is not the root cause.
- PC GUI currently sorts Event Stream by `TS_US` and displays non-negative adjacent `EVENT GAP`; it is not a signed CH2-CH1 pairwise delta display.
- SRAM4 diagnostic evidence before this fix showed firmware-side CH1/CH2 event timestamps differing by only several microseconds, so the small delta was already generated by firmware.
- Diagnostic trigger/peak positions were near the DMA half-frame beginning. This indicates the detector accepted carrier that was already active at frame start as a new trigger, then timestamped the first carrier peak/valley in that frame. That collapses envelope timing into 40k carrier phase timing, producing a few microseconds instead of the envelope interval seen on the oscilloscope.

Fix:

- Added `PRE_TRIGGER_SILENCE_SAMPLES` as a pre-trigger quiet window.
- Added `Has_PreTrigger_Silence()` to reject transitions that do not have quiet samples before the detected point.
- Added `prev_tail_active[ADC_NCH]` and per-frame tail energy detection, so if the previous half-frame ended active, the next frame cannot treat frame-head carrier as a fresh envelope trigger.
- Trigger qualification now requires previous-frame tail inactive, current-frame pre-trigger silence, and sustain-window verification.
